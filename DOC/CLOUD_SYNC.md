# محرك المزامنة السحابية (Cloud Sync Engine)

> المصدر: [`src/services/cloudSync.ts`](../src/services/cloudSync.ts) · الاختبارات: [`cloudSync.test.ts`](../src/services/cloudSync.test.ts) · الجداول: [`supabase/migrations/`](../supabase/migrations/)

كل مخزن Pinia في المنصة يحفظ حالته محليًا (localStorage). هذا المحرك يضيف **طبقة مزامنة سحابية (Supabase) فوق الحفظ المحلي دون تغيير أي واجهة مخزن** — الصفحات والاختبارات لا تعرف بوجوده.

## الصورة الكبيرة

```
┌────────────────┐  watch(source)   ┌─────────────────┐  upsert مؤجل   ┌──────────────┐
│  مخزن Pinia    │ ───────────────► │  cloudSync      │ ─────────────► │   Supabase   │
│  (حالة محلية)  │ ◄─────────────── │  (المحرك)       │ ◄───────────── │  jsonb + RLS │
└────────────────┘   apply(دمج)     └─────────────────┘  جلب الإقلاع   └──────────────┘
        │                                    │
        ▼                                    ▼
   localStorage                    status: off|saving|synced|error
  (يعمل دومًا حتى بلا سحابة)          (شارة اختيارية في الواجهة)
```

- **بلا مفاتيح بيئة** → المحرك خامل (`off`) والمنصة محاكاة كاملة. الاختبارات تعمل هنا دومًا.
- **بمفاتيح** → جلب عند الإقلاع، ورفع مؤجل (debounce 1.2s) عند كل تعديل، وحارس ارتداد يمنع إعادة رفع ما وصل للتو، **وبثّ لحظي** يطبّق تغييرات الأجهزة الأخرى فورًا.

## البثّ اللحظي (Realtime)

بعد الإقلاع يشترك المحرك في `postgres_changes` لصفّ المستند، فأي تغيير من جهاز آخر يصل ويُطبَّق فورًا عبر نفس دالة `apply` — دون إعادة تحميل.

```
جهاز أ ──تعديل──► upsert ──► Supabase ──بثّ postgres_changes──► جهاز ب ──► apply() فورًا
                                    (RLS يصفّي: كل جهاز يستقبل صفوفه فقط)
```

- **التفعيل**: ترحيل `0005` يضيف الجدولين إلى نشرة `supabase_realtime` + `replica identity full`.
- **اسم القناة فريد لكل مستند**: `sync:<table>:<channelId>` — إذ تتشارك المخازن الخاصة الجدول (`account_states`) والفلتر (`owner_id`) نفسيهما، والاسم المكرّر يجعل supabase-js يعيد القناة ذاتها فيرفض إضافة معالج بعد `subscribe()`. لذا `channelId` = مفتاح المخزن (خاص) أو `profile:<slug>` (عام).
- **حارس الصدى**: بثّ الكتابة يعود لصاحبها؛ يُتجاهل بمطابقة `JSON.stringify(incoming) === snapshot()` (كتابتنا أو تغيير بلا أثر). ثم `applyingRemote` يمنع إعادة الرفع.
- **RLS على البثّ**: القناة تُصادَق بتوكن الجلسة، فلا يستقبل الجهاز إلا صفوفه (الخاص) أو الصفحات العامة.
- **تفاعل الجلسة**: الدخول يعيد الجلب والاشتراك؛ الخروج يزيل القناة.
- **مقايضة مقبولة**: قناة لكل مخزن خاص (≈قناة/مخزن على اتصال WebSocket واحد) بدل قناة مشتركة موزِّعة — أبسط وأكثر عزلًا؛ يمكن دمجها في قناة واحدة لاحقًا إن لزم التوسّع.

## الوضعان

| | `syncPrivateDoc` | `syncPublicProfileDoc` |
|---|---|---|
| الجدول | `account_states` | `public_profiles` |
| المفتاح | `(owner_id, store)` | `slug` |
| القراءة | صاحب الصف فقط (RLS) | عامة (الصفحة علنية بطبيعتها) |
| بلا جلسة | **لا مزامنة إطلاقًا** — الحالة محلية و`off` | يقرأ ويكتب الصفوف غير المُدّعاة (ديمو) |
| بجلسة | كل شيء باسم `auth.uid()` | الحفظ **يدّعي** الصف (`owner_id`) فيصير حكرًا على صاحبه |

## عقد المخزن (Contract)

المخزن يقدّم ثلاث دوال فقط — المحرك لا يعرف شكل الحالة، والمخزن لا يعرف الشبكة:

```ts
const { status: syncStatus } = syncPrivateDoc({
  store: 'wallet',                                  // معرّف المستند في account_states
  snapshot: () => ({ txns: txns.value, methods: methods.value }), // ما يُرفع
  apply: (incoming) => { /* دمج بنفس مطبِّع localStorage */ },     // ما يُدمج
  source: [txns, methods],                          // ما يُراقب (بعمق)
})
```

### قواعد `apply` الذهبية
1. **استخدم نفس مطبِّع localStorage** (مثل `mergeStored` في PublicProfileStore) — النسخة السحابية قد تكون من إصدار أقدم من الواجهة وتحتاج ترقية الحقول الجديدة.
2. **تحقق من الشكل قبل الإسناد** (`Array.isArray` / وجود الحقل) — لا تثق بمستند خارجي ثقة عمياء.
3. `apply` يشغّل `watch` مرة — المحرك يبتلعها بحارس الارتداد؛ لا تعالج ذلك بنفسك.

## ربط مخزن جديد — ثلاث خطوات

1. لا شيء في القاعدة — جدول `account_states` يستوعب أي مخزن جديد كما هو.
2. في المخزن: استدعاء `syncPrivateDoc` بالعقد الثلاثي (كما في المثال أعلاه) وتصدير `syncStatus` إن أردت شارة.
3. اختبار المخزن يبقى كما هو (المحرك `off` في vitest) — منطق المحرك نفسه مُغطى في `cloudSync.test.ts` بعميل زائف قابل للحقن (`client`).

## الرسائل المباشرة (تسليم بين المستخدمين)

بخلاف كل ما سبق (مزامنة مستند خاص لكل مستخدم)، الرسائل المباشرة **علائقية**: مستخدم يكتب صفًّا موجّهًا لآخر فيصله لحظيًا. طبقة منفصلة عن محرك المزامنة.

- **الجدول** `direct_messages` (ترحيل 0006): `sender_id, recipient_id, sender_name, recipient_name, body, created_at, read_at`.
- **RLS**: القراءة لطرفَي الرسالة؛ الإدراج باسم صاحب الجلسة فقط (`sender_id = auth.uid()` — لا انتحال)؛ التعليم مقروءًا للمستقبِل.
- **الخدمة** [`directMessages.ts`](../src/services/directMessages.ts): `sendDirectMessage` · `fetchMyMessages` · `markThreadRead` · `subscribeInbound` (بثّ INSERT حيث المستقبِل = أنا) · `resolveProfileOwner` (يحلّ مالك صفحة من slug — جسر «تواصل معي»). نقية وقابلة للحقن.
- **التكامل** في `MessagesStore`: يجلب رسائلي عند الجلسة، يبني المحادثات بمفتاح الطرف الآخر (`peerId`)، ويشترك في الوارد لحظيًا (+إشعار). `send()` على محادثة `peerId` يسلّم فعليًا؛ وإلا سلوك المحاكاة. منع التكرار بمعرّف الرسالة (`mid`).
- **الجسر**: «تواصل معي» في الصفحة العامة — زائر مسجّل يراسل صفحة مُدّعاة لمستخدم آخر → تسليم حقيقي؛ وإلا محاكاة محلية (ديمو غير مُدّعى).
- **تحقق ميداني**: مستخدمان حقيقيان — A يرسل لـB تحت RLS، B يستقبل **لحظيًا بلا إعادة تحميل** (+إشعار)، وانتحال A لهوية B كمرسِل **مرفوض** (42501).

## نموذج الأمان (RLS)

| الجدول | select | insert / update |
|---|---|---|
| `account_states` | `owner_id = auth.uid()` | `owner_id = auth.uid()` |
| `public_profiles` | `true` (عام) | غير مُدّعى (`owner_id is null`) للجميع؛ مُدّعى لصاحبه فقط |
| `direct_messages` | طرفا الرسالة | إدراج: `sender_id = auth.uid()`؛ تحديث: `recipient_id = auth.uid()` |

مفتاح `anon/publishable` علني بطبيعته — **الأمان كله من هذه السياسات**، وقد اختُبرت ميدانيًا (كتابة مجهولة على صف مملوك = صفر صفوف).

## سجل قرار: لماذا «مزامنة مستند» لا جداول علائقية؟

**القرار** (2026-07-05): حالة كل مخزن تُخزن مستند jsonb واحدًا، لا صفوفًا علائقية لكل معاملة/رسالة.

**المسوغات**: المخازن أصلًا «حالة-وثيقة» (تُقرأ وتُكتب كاملة من localStorage)؛ الربط بثلاثة أسطر لكل مخزن؛ ولا حاجة حالية لاستعلامات جزئية أو تقارير خادمية.

**الكلفة المقبولة**: آخر كاتب يكسب (لا دمج حقلي)، ولا فهرسة داخل المستند.

**مسار الخروج**: عند الحاجة لعلائقية مورد ما (مثلًا معاملات المحفظة لتقارير خادمية) يُرحَّل ذلك المورد وحده: جدول مخصص + تكييف `snapshot/apply` عبر طبقة API (انظر `api/openapi.yaml`) — دون المساس ببقية المخازن.

## دلالات الحالة `syncStatus`

| القيمة | المعنى |
|---|---|
| `off` | لا مزامنة: بلا مفاتيح، أو مستند خاص بلا جلسة، أو بعد تسجيل الخروج |
| `saving` | تعديل بانتظار الرفع (داخل مهلة التجميع) أو جلب الإقلاع جارٍ |
| `synced` | آخر جلب/رفع نجح |
| `error` | فشل شبكة أو رفض سياسات — الحالة المحلية سليمة دومًا |

## حدود معروفة (مقصودة)

- لا حل تعارضات بين أجهزة تكتب معًا — آخر كاتب يكسب (مقبول لحساب فردي).
- المستند يُرفع كاملًا كل مرة — حجم حالة المخازن الحالية صغير فلا أثر عمليًا.
- `avatarImage` (dataURL) يضخّم مستند الصفحة — خطته الانتقال إلى Supabase Storage.
